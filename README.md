# Weekly AI-Automation Part-Time Job Report (n8n)

An n8n workflow that runs **twice a week (Mon & Thu mornings)**, pulls remote **AI / automation**
job listings from free job APIs, keeps only **part-time / contract / freelance** roles,
compiles them into an **Excel (.xlsx)** file, and **emails the report** to you (with a
"no matches this week" email on quiet weeks).

Built as a portfolio piece. To turn it into a client deliverable later, only the
recipient email needs to change (see [Reuse for a client](#reuse-for-a-client)).

## What it does

```
Schedule — Mon & Thu 08:00
        │
      Config (search term, filters, recipient)
        ├────────────┬────────────┐
     Remotive      Jobicy      JSearch        ← three job APIs, run in parallel
        └──────┬─────┴─────┬──────┘
            Merge Sources (append)            ← barrier: waits for ALL three to finish
                │                                (without it the Code node fires on the
          Compile Jobs (Code)                    first arrival and can't see the others)
                │                              ← normalize + filter part-time/contract/freelance
          Has Results?  (IF)                     + automation keyword + dedupe + sort
            ├── true (empty) ── Email No Results
            └── false ──────── To XLSX ── Email Report (with .xlsx attached)
                          └──── Store in Log (n8n Data Table)
```

- **Sources:** [Remotive](https://remotive.com), [Jobicy](https://jobicy.com), and
  **[JSearch](https://rapidapi.com/letscrape-6bRBa3QguO5/api/jsearch)** (RapidAPI). Remotive
  and Jobicy are free/no-key and filter by "automation" server-side (Remotive asks for
  **≤ 4 API calls/day** — the weekly schedule respects that; don't spam it while testing).
  JSearch aggregates **Indeed / LinkedIn / Glassdoor / ZipRecruiter / Upwork** via Google for
  Jobs and needs a free RapidAPI key in a **Header Auth** credential (see setup). It calls the
  **`/search-v2`** endpoint (the old `/search` now 404s) with `employment_types=PARTTIME,CONTRACTOR`
  and `remote_jobs_only=true`.
- **Merge barrier:** the three sources fan into a **Merge** node (append, 3 inputs) before the
  Code node. This is required — n8n runs a multi-input Code node as soon as the *first* branch
  arrives, so without Merge `$('Jobicy')`/`$('JSearch')` throw "hasn't been executed" and those
  sources silently contribute nothing.
- **Filter:** an employment type of `part_time`, `contract`, or `freelance`, **and** an
  automation/AI keyword somewhere in the title/tags/description, posted within 30 days.
- **Output:** `ai-automation-part-time-jobs.xlsx` with columns
  `source, title, company, job_type, location, salary, posted_date, url, tags`.

## Setup (one time)

1. **Import the workflow.** In n8n: *Workflows → ⋯ → Import from File →* `workflow.json`.
2. **Add a Gmail credential.** *Credentials → New → "Gmail OAuth2 API"* and complete the
   Google sign-in ([n8n Gmail credential docs](https://docs.n8n.io/integrations/builtin/credentials/google/)).
3. **Select that credential on both Gmail nodes** — `Email Report` and `Email No Results`.
3b. **Add the JSearch key.** Subscribe (free) to [JSearch on RapidAPI](https://rapidapi.com/letscrape-6bRBa3QguO5/api/jsearch)
   → *Pricing → Basic (Free) → Subscribe*. Then *Credentials → New → "Header Auth"* with
   **Name** = `X-RapidAPI-Key` and **Value** = your key, and select it on the **JSearch** node.
   (The endpoint is `/search-v2`; the deprecated `/search` returns `404 Endpoint '/search' does not exist`.)
4. **Sanity-check two fields on `Email Report`** (n8n versions differ slightly, so confirm
   after import — see [Version notes](#version-notes)):
   - *Email Type* = **HTML**.
   - Under *Options → Attachments*, one attachment with binary property **`data`**
     (that is the property `To XLSX` writes the Excel file to).
5. **Test run.** Open the workflow and click **Execute Workflow**. You should get an email
   within a minute — either the report with the `.xlsx` attached, or the "no matches" note.
6. **Activate.** Toggle the workflow **Active** so the Mon & Thu 08:00 schedule runs it automatically.

## Configure

Open the **Config** node to change behavior without touching code:

| Field | Default | Meaning |
|---|---|---|
| `searchTerm` | `automation` | Query sent to both APIs |
| `maxAgeDays` | `30` | Drop postings older than this |
| `allowedTypes` | `part_time,contract,freelance` | Employment types to keep (comma-separated) |
| `requireKeyword` | `true` | Require an automation/AI keyword in title/tags/description |
| `recipient` | your email | Where the report is emailed (change this per client) |

The keyword list and the title filters live in the **Compile Jobs** Code node because they
are long and rarely change:
- `DEFAULT_CONFIG.keywords` — the automation/AI terms a row must mention.
- `DEFAULT_CONFIG.excludeTitleKeywords` — *soft* exclusions (sales, writer, recruiter, …):
  dropped **unless** the title itself carries an automation keyword (so "Sales Automation
  Engineer" survives).
- `DEFAULT_CONFIG.blockTitleKeywords` — *hard* block (secretary, receptionist, data entry,
  administrative assistant, …): dropped **even if** the title says "automation", to catch
  clerical roles like "Secretary (Office Automation)".

> **Why the keyword gate?** The APIs' `automation` search is loose (it matches job
> descriptions), so it can return the odd "Head of Sales" posting. The keyword gate is a
> quality filter that checks the **description**, not just the title — so it keeps genuine
> automation roles with generic titles (e.g. "Data Labeling Specialist") while dropping
> unrelated noise. Setting `requireKeyword=false` relies on the API search alone.

## Reuse for a client

Change the recipient in exactly one place: the **`recipient`** field on the **Config** node.
Both Gmail nodes read it via `{{ $('Config').first().json.recipient }}`, so nothing else
needs touching. (To run several clients, duplicate the workflow and change that one field,
or drive it from a list.)

## Security — no secrets in this repo, bring your own credentials

**This repository contains no secrets.** There are no API keys, tokens, or passwords in any
file — not in `workflow.json`, not in the scripts, nowhere. To run it you supply your own:

- **Gmail** → an *OAuth2* credential you create in n8n (browser consent).
- **JSearch** → a free RapidAPI key stored in a *Header Auth* credential (`X-RapidAPI-Key`).
- **Remotive / Jobicy** → no key needed.

How secrets are kept out:

- The workflow references credentials by **n8n's credential system**, never as text in a node.
  (n8n's own rule: a token in a Set/HTTP field is a leak — so the RapidAPI key lives in a
  Header Auth credential the node points at, and is never serialized into `workflow.json`.)
- The helper scripts read the n8n endpoint/token from **environment variables**
  (`N8N_MCP_URL`, `N8N_MCP_TOKEN`, `N8N_API_KEY`) — nothing is hardcoded.
- **`.gitignore`** excludes `.tmp/` (run/debug data and instance IDs), `node_modules/`,
  `.env*`, `credentials.json`, `token.json`, and key/cert files.
- The email recipient and instance URL in these files are **placeholders**
  (`you@example.com`, `YOUR-INSTANCE.app.n8n.cloud`) — set your own after import.

If you fork this, keep that discipline: pass secrets via env vars or the n8n credential store,
and never commit `.env`, `credentials.json`, or `token.json`.

## Deploying

Two ways to get this onto a live instance:

**A) Import the file** — drag `workflow.json` into n8n (Workflows → Import from File).

**B) Push programmatically** — used for the current deployment. Set the endpoint + token,
then generate the SDK code, validate, and create:
```bash
export N8N_MCP_URL="https://<instance>.app.n8n.cloud/mcp-server/http"   # n8n → MCP access
export N8N_MCP_TOKEN="<bearer token>"
node scripts/build-sdk-code.js                                          # -> workflow.sdk.ts
node -e 'require("fs").writeFileSync("_args.json",JSON.stringify({code:require("fs").readFileSync("workflow.sdk.ts","utf8")}))'
cat _args.json | node scripts/mcp-call.js validate_workflow -          # must be valid:true
cat _args.json | node scripts/mcp-call.js create_workflow_from_code -  # returns workflow id + url
```
`scripts/mcp-call.js` is a tiny stateless client for n8n's native MCP server (also supports
`:list`, `get_workflow_details`, `publish_workflow`, `execute_workflow`, `update_workflow`,
`create_data_table`, …). The token is a secret — pass it via env, never commit it.
`scripts/deploy-to-n8n.js` is an alternative that uses the public REST API (`X-N8N-API-KEY`)
for instances where that API is enabled.

`workflow.json` (12 nodes) is the current source of truth and matches the live workflow;
`workflow.sdk.ts` is the initial-deploy snapshot (9 nodes). Later changes — recipient
parameterization, the relevance filter, the Data Table log node, the JSearch source
(`/search-v2` + Header Auth), and the Merge sync barrier — were applied to the live workflow
with `update_workflow` and folded back into `workflow.json` via `scripts/build-workflow.js`.

`src/normalize-core.js` is the single source of truth for the transform; `normalize-jobs.js`
inlines the same functions (n8n Code nodes can't `require` local files) and adds the n8n
glue. Keep them in sync — `scripts/test-workflow-code.js` guards the embedded copy.

## Version notes

`workflow.json` targets these node versions: `scheduleTrigger` 1.2, `set` 3.4,
`httpRequest` 4.2, `merge` 3.2, `code` 2, `if` 2.2, `convertToFile` 1.1, `gmail` 2.1,
`dataTable` 1.1.

**These were validated against the real `n8n-nodes-base` 2.28.x schemas** (see
`scripts/validate-node-schemas.js`): every node's typeVersion is supported, and the
nested shapes were confirmed — Gmail `emailType` accepts `html`, the attachment binding is
`options.attachmentsUi.attachmentsBinary[].property`, `convertToFile` supports the `xlsx`
operation with `fileName`/`sheetName`, and `if` v2.2 uses the `filter` conditions type. So
step 4's checks should already be correct on n8n 2.28+; just confirm them if you run an
older/newer major. (A full local *execute* wasn't possible — the n8n 2.28.5 **server** has
an unrelated module-loading bug on Windows — but the transform itself is proven end-to-end
by `scripts/test-workflow-code.js` running the exact Code-node body against the live APIs.)
